## ⚠️ 执行规则

### 规则 0: Ritual First（最重要）
**所有涉及写代码的 task，首选用 `start_ritual` 执行。** Ritual 自动走完 design → review → implement → verify 全流程。

只有在 ritual 真的失败（报错、卡住、环境问题）时，才降级为手动流程。

### 手动 Fallback 流程（仅当 ritual 失败时）
如果 ritual 不可用或反复失败，按以下步骤**严格顺序**手动执行：

1. **Requirements 文档**
   - 写 `.gid/features/<feature>/requirements.md`（遵循 draft-requirements skill）
   - Review 第 1 轮：spawn review sub-agent → 写 findings → **apply ALL findings**
   - Review 第 2 轮：再 spawn review sub-agent → 写 findings → **apply ALL findings**
   - 两轮 review 全部 apply 后，requirements 才算完成

2. **Design 文档**
   - 写 `.gid/features/<feature>/design.md`
   - Review 第 1 轮：spawn review sub-agent → 写 findings → **apply ALL findings**
   - Review 第 2 轮：再 spawn review sub-agent → 写 findings → **apply ALL findings**
   - 两轮 review 全部 apply 后，design 才算完成

3. **Graph + Tasks**
   - `gid_design` 从 design 文档生成 graph nodes + task nodes + feature nodes
   - `gid_advise` 检查 graph 质量
   - Review tasks（review-tasks skill）→ apply findings

4. **Implement**
   - 按 graph 中 task 依赖顺序逐个实现
   - 每个 task 完成后 `gid_complete`

5. **Test + Commit**
   - `cargo test` 全 pass
   - commit with descriptive message

**关键：不跳步。** Requirements 没过两轮 review 不写 design。Design 没过两轮 review 不建 graph。Graph 没建好不写代码。
