## 🫀 Layer 3 — Adaptive Behavior Modulation (15:52-16:10 ET)

### What was built
Interoceptive Layer 3: self-correction loop. System now detects deviations from its own baselines and injects behavioral directives.

### Architecture
```
Layer 1 (signal collection) → Layer 2 (aggregation) → Layer 3 (action)
                                                         ↓
                              ┌─ Adaptive Baselines (σ-deviation)
                              ├─ Cold-start Fallback (hardcoded thresholds)
                              ├─ System Prompt Directives (cross-request)
                              └─ Mid-loop Intervention (within-request)
```

### Changes

**engramai:**
- `types.rs`: NEW `AdaptiveBaseline` — Welford's online algorithm for rolling mean+stddev, optional exponential decay for drift adaptation. `DeviationLevel` enum (Uncalibrated/Normal/Elevated/High/Extreme)
- `hub.rs`: Baselines tracked per (source, domain) pair. `baseline()`, `deviation_level()`, `is_domain_calibrated()` query methods
- `regulation.rs`: REWRITTEN — `evaluate_with_hub()` uses adaptive σ-thresholds. Fallback to conservative hardcoded values during cold start (<20 samples). Regulation messages include σ-deviation info
- 36 interoceptive tests pass (was 26)

**RustClaw:**
- `agent.rs`: 
  - NEW `active_regulation: Arc<RwLock<Vec<RegulationAction>>>` on AgentRunner
  - NEW `format_regulation_directives()` — converts actions to system prompt section
  - Post-loop: feeds signals → evaluates regulation → stores active actions
  - System prompt injection: regulation directives appear after interoceptive state
  - Mid-loop: 3+ consecutive failures → injects "[SYSTEM: change strategy]" message
- `interoceptive.rs`: `consecutive_failures()` accessor on ExecutionStressMeter
- `memory.rs`: `interoceptive_regulate()` now uses `evaluate_with_hub()` with adaptive baselines
- 247 tests pass (unchanged count)

### Key design decisions
- **Self-adaptive, not hardcoded**: σ-based deviation from own baseline, not fixed thresholds
- **Cold-start safety**: first 20 samples use conservative fallbacks (e.g. -0.5 valence, 20% success)
- **Two timescales**: cross-request (system prompt directives) + within-request (mid-loop intervention)
- **Welford's algorithm**: O(1) per sample, no stored history needed, numerically stable

## F2-B6 Voice Emotion Analysis — COMPLETED

### What was built
Speech Emotion Recognition (SER) pipeline that detects emotional tone from voice messages and feeds it into the interoceptive system.

### Architecture
```
Voice msg (OGG) → ffmpeg → WAV (16kHz mono)
                              ├→ whisper-cli (STT) → transcription text
                              └→ voice_emotion.py (SER) → emotion signal
                                   ↓
                              InteroceptiveHub ("communication" domain)
                                   ↓
                              System prompt: "[🎙️ Voice tone: angry (90%)]"
```

### Components

1. **engramai** (`interoceptive/types.rs`):
   - New `SignalSource::VoiceEmotion` + `SignalContext::VoiceEmotion`
   - DomainState::update() handles VoiceEmotion (valence_trend + anomaly_level)

2. **Python SER** (`scripts/voice_emotion.py`, ~120 lines):
   - Model: wav2vec2-lg-xlsr-en-speech-emotion-recognition (8 emotions)
   - stdlib `wave` for audio loading, ~3s inference on M-series

3. **Rust module** (`src/voice_emotion.rs`, ~250 lines):
   - Russell's circumplex model: emotion → (valence, arousal)
   - Confidence-weighted, 20% threshold filter
   - 15s timeout, graceful failure (None on any error)

4. **Telegram integration** (`src/channels/telegram.rs`):
   - STT + SER parallel via `tokio::join!` (zero added latency)
   - Signal fed to InteroceptiveHub "communication" domain
   - `[🎙️ Voice tone: emotion (XX%)]` prefix on transcribed message
