## 💬 engram Synthesis 性能优化讨论 (23:22-23:30 ET)

potato 和我深入讨论了 synthesis 聚类的性能优化和未来架构。

### 问题诊断
- 13k memories synthesis 耗时 443 秒
- 瓶颈 1: 数据加载 — hebbian_links 和 memory_entities 逐条 SQL 查询 (26k queries)
  - hebbian_links: 44,068 条总记录，仅 264 条 strength>0，涉及 66 个 distinct memory IDs
  - memory_entities: 4,283 条
- 瓶颈 2: k-NN 图构建 O(n²) — 13k² = 1.7亿次 compute_weight 调用

### Root Fix 方案
1. **数据加载**: 新增 bulk 方法 `get_all_hebbian_links()` + `get_all_memory_entities()`，各 1 次查询
2. **k-NN Two-Stage Pipeline**:
   - Stage 1a: Embedding ANN (VP-tree) 找 top-k' 候选, O(n log n)
   - Stage 1b: Hebbian injection — 直接注入已知关联（embedding 距离无关）
   - Stage 2: MultiSignal 精确计算只跑 candidate pairs
   - 加速: 13k=280x, 100k=2200x
3. **ANN 实现**: VP-tree 先行（纯 Rust 无依赖），百万级再换 HNSW

### Infomap Incremental 问题
- Infomap 数学上不能 incremental（map equation 全局优化）
- 解决: Hot/Warm/Cold 三层架构（类 LSM-tree）:
  - Hot: 新记忆 ANN 找邻居 → 分配到最近 community, O(log n)
  - Warm: 每天 affected communities local re-cluster
  - Cold: 每周全量 Infomap

### 百万级多 Agent 记忆共享架构
- 估算: 10 agents × 350/天 → 一个月百万
- **核心洞察: 不该把百万记忆放进同一个 Infomap**
- Hierarchical Memory Architecture:
  - Level 0: 每个 agent 本地记忆 → 本地 Infomap (10k-100k)
  - Level 1: 共享层 — 交换 cluster centroids（不是原始记忆）
  - Level 2: Meta-Infomap 在 centroids 上跑（几百-几千节点）
  - Cross-agent Hebbian: 跨 agent co-recall 关联
  - 隐私天然隔离：只共享 centroid embedding + 标签
