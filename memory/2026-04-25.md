## ISS-019 Root Fix — Part 3: Orphan Sweep ✅

**Completed all 3 parts of the ritual cancel persistence root fix.**

### Part 3 details (just landed)
Added orphan sweep that runs once at daemon startup. Detects zombie ritual state files (non-terminal phase + dead `adapter_pid`, OR no pid + stale >24h) and rewrites them as `Cancelled` via the canonical `transition(state, UserCancel)` so the result is indistinguishable from a user-driven cancel. Preserves the original dead PID on disk for forensics (doesn't overwrite with sweeper's pid).

**Files touched:**

- `src/channels/telegram.rs` — runs sweep in `start()` before main bot loop, logs swept zombies as warnings

**Tests:**

- 336/336 rustclaw bin tests pass total
- 48/48 gid-core state machine tests still green

- status: missing field (legacy file → deserializes as Active)

### Design choices worth remembering
- **Free function `sweep_orphans_in(&Path)`** instead of method-only — makes it testable without constructing a full `RitualRunner` (which needs an LLM client). Method `RitualRunner::sweep_orphans()` is a thin wrapper.
- **Preserve dead PID on disk** — sweep writes via direct `serde_json::to_string_pretty + fs::write`, NOT via `save_state` which would stamp the sweeper's PID. Dead PID is forensic data.
- **Stale threshold = 24h** for no-pid files — protects against racing pre-PID-stamping runners mid-write.
- **PID reuse caveat documented** — long-dead pid reassigned to unrelated process reads as alive. Acceptable because (a) sweep is best-effort, (b) PID space is large, (c) sweep frequency is bounded.
- **Drive through FSM, not direct field mutation** — `transition(state, UserCancel)` ensures status, transitions log, and any future state machine invariants are all satisfied uniformly.
