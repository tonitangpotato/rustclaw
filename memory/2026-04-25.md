**Context**: While refactor-deleting planned code nodes from `.gid/graph.db` via `gid_refactor` tool, discovered orphaned edges remained — `BatchOp::DeleteNode` did not delete incident edges.

### Decision (potato + me)
- ✅ Root fix, not patch. `BatchOp::DeleteNode` semantics = remove vertex AND all incident edges.
- ✅ Use option (i): point RustClaw `Cargo.toml` at local `gid-rs` (path dependency) — avoid publish cycle.

### Fix shipped (gid-rs)
- File: `crates/gid-core/src/storage/sqlite.rs`
- Both call sites patched (`execute_batch` + `execute_migration_batch`):
  ```rust
  BatchOp::DeleteNode(id) => {
      tx.execute("DELETE FROM edges WHERE from_node = ? OR to_node = ?", params![id, id])?;
      tx.execute("DELETE FROM nodes WHERE id = ?", params![id])?;
  }
  ```
- 3 regression tests added: `test_iss037_delete_node_in_execute_batch_removes_incident_edges`, `..._migration_batch_...`, `..._self_loop_removed`. All pass.
- Full suite: **690/690 pass**, 0 new warnings.
- ISS doc: `gid-rs/.gid/issues/ISS-037-deletenode-orphans-edges.md`

### Current orphan edges in rustclaw `.gid/graph.db`
- 87 orphan edges, **all pointing to a missing `ISS-021` node** (parent issue, pre-existing data drift — NOT from today's delete operation).
- Cleanup query (run post-restart):
