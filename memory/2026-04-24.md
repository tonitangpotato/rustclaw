**选型决策 (GUARD-11)**: Option A — 设计对齐代码事实，而不是反过来。

### 方法论固化 (Incremental Output Protocol)

之前遇到的根因：单次写 500+ 行文档容易 connection 断，且丢失中间进度。

**两层固化：**

1. **Skill 层** (v1.0.0 → v1.1.0)
   - skills/review-design/SKILL.md
   - skills/review-requirements/SKILL.md
   - skills/review-tasks/SKILL.md
   
   三个 review skill 都加了 "Incremental Output Protocol" 章节，规定：skeleton 先写一次 (~40 行)，每个 FINDING 用单独的 edit_file 调用 append。

2. **AGENTS.md 层** — 新增 Rule 3b "Skeleton-First Append Pattern"
   - 是 Rule 3 (Incremental Write Pattern) 的具体化变种
   - 适用于结构化列表文档（review、任务清单、catalog 等）
   - 核心：one item = one tool call；每次 edit 都是 durable checkpoint

### 为什么要加 Rule 3b 而不是只更新 skill

Rule 3 是通用 "大输出分多次写" 的建议，但没说**怎么切**。Rule 3b 明确了结构化列表文档的具体模式：skeleton + per-item append + marker-carry。下次写任何有重复 item 的长文档（不一定是 review），都可以直接走这个模式。
