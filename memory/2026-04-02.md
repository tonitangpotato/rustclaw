
## GID 生态架构决策（深夜讨论）

### 四个项目的最终定位
- **gid-core** — 图引擎 + 共享类型定义（事件格式、状态 schema）
- **gid-harness** — AI 自主开发的执行引擎，跑在 RustClaw 内部
- **gidterm** — TUI surface，纯展示层 + 人类控制面板，读 execution-log.jsonl
- **agentctl** — daemon 进程管家，管 RustClaw/OpenClaw 等长期进程

### 关键决策
1. **选了方案 B**：harness 独立写，gidterm 变成纯 UI
   - gidterm 的 executor/scheduler 代码会废弃，但设计经验 inform harness
   - PTY 执行 shell 和 spawn LLM sub-agent 差异太大，不值得硬抽 trait
2. **文件系统是 backend** — 不需要 server 进程
   - graph.yml（task 定义+状态）
   - execution-log.jsonl（append-only 事件流）
   - execution-state.json（当前运行状态）
3. **所有 surface 读写同一套文件** — Telegram、gidterm、CLI 都是 surface
4. **共享的是协议不是代码** — 事件格式和状态 schema 定义在 gid-core 里

### 7-Phase 流程
- Phase 1-3: 人机协作（idea → requirements → design）
- Phase 4-7: AI 自动（planning → execution → verification → integration）
- mixed mode: Phase 4 生成 plan 后 gate → 人 approve → 自动执行
- approval gate 用 tag 实现：`gate:human` 标签

### 人类控制机制
- Telegram inline buttons: approve/reject/pause
- gidterm: TUI 实时 DAG 渲染 + 操作
- /stop → .gid/pause flag file
- tag-based gate: 关键 task 标 gate:human
