## Bug Fix: fd leak from notify kqueue backend (22:30 ET)

### Problem
RustClaw would accumulate ~200+ file descriptors over time, eventually hitting "Too many open files" (os error 24). This caused:
- All LLM API requests to fail
- exec tool commands to fail
- ffmpeg/whisper-cli to fail (looked like "not installed")
- Bot appeared dead — typing indicator but no response

### Root Cause
`notify` crate was configured with `macos_kqueue` backend in Cargo.toml:
```toml
notify = { version = "7", default-features = false, features = ["macos_kqueue"] }
```

kqueue opens **one fd per watched file/directory entry**. The config watcher watched the workspace root (`/Users/potato/rustclaw/`), and kqueue opened fds for every file in the directory — including `target/release/.fingerprint/` (200+ build artifact files).

### Fix
1. **Switched to FSEvents backend**: `features = ["macos_fsevent"]` — macOS native event system, no per-file fds
2. **Watch config file directly** instead of parent directory (defense in depth)

## Feature: Skill Trigger System (22:57 ET)

### What was built
Implemented **dynamic skill injection** — skills are no longer always loaded into the system prompt. Instead, SKILL.md files now support YAML frontmatter with metadata:

```yaml
---
name: Idea Intake Pipeline
description: Process incoming ideas/links
triggers:
  - http://
  - https://
  - idea:
  - 想法:
priority: 50
always_load: false
max_context_bytes: 4096
---
```

### How it works
1. **`parse_skill_frontmatter()`** — Parses YAML frontmatter from SKILL.md, extracts metadata (name, description, triggers, priority, always_load, max_context_bytes). No serde_yaml dependency — hand-rolled parser.
2. **`Workspace.match_skills()`** — Matches skills against user message via case-insensitive substring matching of triggers. `always_load` skills always included. Results sorted by priority.
3. **`build_system_prompt_with_skills()`** — New method that accepts `user_message: Option<&str>` for dynamic injection. All 3 call sites in agent.rs updated.
4. **Backward compatible** — Skills without frontmatter still work (defaults: priority=100, no triggers, not always_load).
5. **15 new tests** covering frontmatter parsing + trigger matching (155 total tests, all pass).

### Files Changed
- `src/workspace.rs` — SkillMetadata struct, parse_skill_frontmatter(), match_skills(), build_system_prompt_with_skills(), 15 unit tests
- `src/agent.rs` — 3 call sites updated to pass user message
- `skills/idea-intake/SKILL.md` — Added frontmatter metadata

### Build
- `cargo test` ✅ (155 tests pass, 0 failures)
- `cargo build --release` ✅

### Why kqueue was there
Likely chosen by sub-agent for its precision (can detect rename vs modify vs delete). Overkill for config hot-reload — fsevent's "directory changed" is sufficient.

### When to use kqueue
Only if precise file event types are needed (IDE-like file sync). For config watching, always use fsevent.
